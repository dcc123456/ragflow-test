## 测试用例编写规范

### 文件命名

- 格式：`{功能名}-test.md`
- 示例：`billing-test.md`、`login-test.md`

### 文件结构

```markdown
## 测试任务名称

**测试目标**: 简要说明测试目的

**前置条件**:

- 环境要求
- 账号要求
- 数据要求

**测试前判断**:

- 如果test-script目录下有对应的脚本文件，则执行该脚本文件，如果脚本执行全部成功，则跳过当前测试用例的测试步骤
- 否则执行测试用例/task目录中的测试步骤
  例如：用户要测试登录功能，则执行用户登录脚本（/test-script/login-test.playwright.js），如果脚本执行成功，则跳过登录步骤，否则执行登录功能测试用例（/task/login-test.md）

**测试数据**:
| 数据类型 | 值 | 说明 |
|----------|-----|------|
| URL | http://xxx | 测试地址 |
| 账号 | xxx | 用户名/密码 |
| 文件 | path/to/file | 上传文件路径 |

**执行步骤**:

1. 步骤描述
2. 步骤描述
   - 预期结果
3. 步骤描述
   - 预期结果

**环境清理**:

- 清理内容1
- 清理内容2
```

### 编写要求

| 要求     | 说明                             |
| -------- | -------------------------------- |
| 步骤明确 | 每步只做一件事，避免复合步骤     |
| 结果可判 | 每步有明确的预期结果，可自动判定 |
| 独立性强 | 步骤间无隐式依赖，可单独执行     |
| 参数化   | URL/账号等用变量占位，便于替换   |
| 时效性   | 定期更新账号/URL等过期信息       |

### 步骤描述模板

```
1. **操作描述**
   - 操作：点击/输入/选择/上传...
   - 目标：元素定位器或描述
   - 预期：页面变化或结果
```
